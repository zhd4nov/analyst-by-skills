# Пользовательские истории

Статус: complete

## История
ID: US-01
Название: Создание цели с выбором типа отслеживания
Источник
- Спецификация -> Пользовательские сценарии: пользователь создает цель и выбирает обычную цель со шагами или простую цель.
- Спецификация -> Функциональные требования: система должна позволять пользователю отметить цель как простую при создании.
- Лог уточнений: CL-03.
Актор: пользователь приложения по постановке личных целей.
Потребность: выбрать при создании цели, будет ли цель отслеживаться по шагам или как простая цель.
Ценность: сразу задать подходящий способ отслеживания без лишних шагов для простой цели.
Сценарий: пользователь создает цель и указывает, является ли она простой или будет достигаться через шаги-задачи.
Критерии приемки
- При создании цели пользователь может отметить цель как простую.
- Если цель отмечена как простая, система не требует создания шагов для нее.
- Если цель не отмечена как простая, цель может содержать шаги-задачи для расчета прогресса.
Бизнес-правила
- Простая цель имеет бинарное состояние выполнения вместо процента.
Зависимости
- нет.
Открытые вопросы
- нет.

## История
ID: US-07
Название: Начало достижения цели
Источник
- Спецификация -> Пользовательские сценарии: пользователь вручную начинает цель действием "начать".
- Спецификация -> Пользовательские сценарии: пользователь выполняет первый шаг цели, и цель переходит из `черновик` в `в процессе`.
- Спецификация -> Модель состояний / Переходы: `черновик` -> `в процессе`.
- Лог уточнений: CL-06.
Актор: пользователь приложения по постановке личных целей.
Потребность: перевести цель из черновика в процесс достижения вручную или через первый выполненный шаг.
Ценность: явно отделять еще не начатые цели от целей, по которым уже идет работа.
Сценарий: пользователь нажимает "начать" у цели или выполняет первый шаг; система переводит цель в статус `в процессе`.
Критерии приемки
- Если пользователь нажимает "начать" у цели в статусе `черновик`, цель переходит в статус `в процессе`.
- Если пользователь выполняет первый шаг цели в статусе `черновик`, цель переходит в статус `в процессе`.
- Для цели в статусе `в процессе` применяются правила отображения прогресса.
Бизнес-правила
- Цель переходит из `черновик` в `в процессе` после выполнения первого шага.
- Цель переходит из `черновик` в `в процессе` после ручного действия пользователя "начать".
Зависимости
- цель должна быть в статусе `черновик`.

## История
ID: US-02
Название: Просмотр статуса и прогресса цели
Источник
- Спецификация -> Термины и определения: статус цели и прогресс цели.
- Спецификация -> Функциональные требования: система должна отличать статус цели от прогресса цели.
- Спецификация -> Функциональные требования: для цели со шагами в статусе `в процессе` система должна показывать процент прогресса.
- Лог уточнений: CL-01.
Актор: пользователь приложения по постановке личных целей.
Потребность: видеть статус цели и текущий прогресс, если цель отслеживается по шагам.
Ценность: понимать, на каком этапе находится цель и насколько она продвинута.
Сценарий: пользователь открывает список или карточку цели и видит статус цели; для цели со шагами в процессе видит процент прогресса, для простой цели видит `выполнена` / `не выполнена`.
Критерии приемки
- Система показывает статус цели отдельно от прогресса.
- Для цели со шагами в статусе `в процессе` отображается процент прогресса.
- Процент прогресса отображается целым числом.
- Для простой цели процент не отображается; отображается состояние `выполнена` или `не выполнена`.
- Для завершенной цели отображается статус `завершена`.
Бизнес-правила
- Статус и прогресс цели не должны подменять друг друга.
- Процентный прогресс отображается только для цели со шагами в статусе `в процессе`.
- Процент прогресса всегда отображается целым числом и округляется.
- Простая цель имеет бинарное состояние выполнения вместо процента.
Зависимости
- нет.
Открытые вопросы
- нет.

## История
ID: US-03
Название: Обновление прогресса через шаги цели
Источник
- Спецификация -> Функциональные требования: система должна рассчитывать процент прогресса цели со шагами на основе выполненных шагов.
- Спецификация -> Функциональные требования: система должна позволять пользователю добавлять шаги и удалять невыполненные шаги у цели в статусе `в процессе`.
- Спецификация -> Бизнес-правила: цель в процессе можно редактировать, добавлять шаги и удалять невыполненные шаги.
- Лог уточнений: CL-01; CL-03; CL-04; CL-05.
Актор: пользователь приложения по постановке личных целей.
Потребность: выполнять, добавлять и удалять невыполненные шаги цели в процессе достижения.
Ценность: поддерживать актуальный процент прогресса в соответствии с реальными действиями по цели.
Сценарий: пользователь работает с целью в процессе, отмечает шаги выполненными, добавляет новые шаги или удаляет невыполненные шаги; система пересчитывает процент прогресса и отображает его целым числом.
Критерии приемки
- После выполнения шага процент прогресса цели со шагами пересчитывается и отображается целым числом.
- После добавления шага процент прогресса пересчитывается с учетом нового состава шагов и округляется.
- После удаления невыполненного шага процент прогресса пересчитывается с учетом оставшихся шагов и округляется.
- Выполненный шаг удалить нельзя.
- Добавление и удаление шагов доступны только для цели в статусе `в процессе`.
Бизнес-правила
- Процент прогресса зависит от выполненных шагов внутри цели.
- Процент прогресса всегда отображается целым числом и округляется.
- Цель в процессе можно редактировать, добавлять шаги и удалять невыполненные шаги.
- Выполненный шаг удалить нельзя.
Зависимости
- цель должна быть в статусе `в процессе`.
Открытые вопросы
- нет.

## История
ID: US-04
Название: Подтверждение завершения цели со шагами
Источник
- Спецификация -> Пользовательские сценарии: после выполнения всех шагов пользователь подтверждает завершение цели или добавляет новые шаги.
- Спецификация -> Модель состояний / Переходы: `в процессе` -> `завершена` для цели со шагами.
- Лог уточнений: CL-02.
Актор: пользователь приложения по постановке личных целей.
Потребность: подтвердить, что цель действительно достигнута после выполнения всех шагов.
Ценность: не завершать цель преждевременно, если пользователь хочет добавить дополнительные шаги.
Сценарий: все шаги цели выполнены; система спрашивает пользователя, достигнута ли цель; пользователь подтверждает завершение или добавляет новые шаги.
Критерии приемки
- После выполнения всех шагов система не переводит цель в `завершена` без подтверждения пользователя.
- Если пользователь подтверждает достижение, цель получает статус `завершена`.
- Если пользователь добавляет шаги, цель остается в статусе `в процессе`.
- После добавления шагов процент прогресса пересчитывается.
Бизнес-правила
- Завершение цели со шагами через автоматический переход запрещено без подтверждения пользователя.
- Добавление шагов вместо подтверждения завершения оставляет цель в процессе и меняет расчет прогресса.
Зависимости
- все шаги цели должны быть выполнены.
Открытые вопросы
- нет.

## История
ID: US-05
Название: Ручное завершение простой цели
Источник
- Спецификация -> Функциональные требования: простая цель должна завершаться только вручную пользователем.
- Спецификация -> Модель состояний / Переходы: `в процессе` -> `завершена` для простой цели.
- Лог уточнений: CL-03.
Актор: пользователь приложения по постановке личных целей.
Потребность: самостоятельно завершить простую цель без шагов.
Ценность: закрывать цели, которым не нужен детальный прогресс по задачам.
Сценарий: пользователь открывает простую цель и вручную отмечает ее завершенной.
Критерии приемки
- Простая цель не завершается автоматически.
- Пользователь может вручную завершить простую цель.
- После ручного завершения простая цель получает статус `завершена`.
- Для простой цели не отображается процент прогресса.
Бизнес-правила
- Простая цель завершается только ручным действием пользователя.
- Простая цель имеет бинарное состояние выполнения вместо процента.
Зависимости
- цель должна быть простой.
Открытые вопросы
- нет.

## История
ID: US-06
Название: Просмотр завершенных целей в достижениях
Источник
- Спецификация -> Функциональные требования: завершенная цель должна отображаться в списке достижений и не отображаться в списке активных целей.
- Спецификация -> Ограничения: завершенная цель является терминальной и не может быть возвращена назад.
- Лог уточнений: CL-02; CL-03.
Актор: пользователь приложения по постановке личных целей.
Потребность: видеть завершенные цели отдельно от активных.
Ценность: отличать текущие цели от уже достигнутых результатов.
Сценарий: пользователь завершает цель и затем видит ее в списке достижений, а не в активных целях.
Критерии приемки
- После завершения цель исчезает из списка активных целей.
- После завершения цель появляется в списке достижений.
- Завершенную цель нельзя вернуть в статус `черновик` или `в процессе`.
Бизнес-правила
- Завершенная цель исключается из списка активных целей и отображается в списке достижений.
- Статус `завершена` является терминальным.
Зависимости
- цель должна быть завершена.
Открытые вопросы
- нет.
