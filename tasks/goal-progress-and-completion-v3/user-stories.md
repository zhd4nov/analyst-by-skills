История
ID: US-01
Название: Просмотр состояния и прогресса цели
Источник
- Пользовательский сценарий: пользователь открывает список целей и видит статус и прогресс каждой цели
- Функциональное требование: система должна показывать статус цели в списке целей и в карточке цели
- Функциональное требование: система должна показывать вычисляемый процент прогресса по цели
Актор: Пользователь приложения по постановке личных целей
Потребность: видеть текущий статус и прогресс цели
Ценность: понимать, на каком этапе находится цель и есть ли по ней реальное продвижение
Сценарий: пользователь открывает список целей или карточку цели и видит актуальный статус и вычисляемый прогресс
Критерии приемки
- в списке целей отображается текущий статус каждой цели
- в списке целей отображается текущий вычисляемый процент прогресса
- в карточке цели отображаются статус, процент прогресса и состояние шагов, если шаги существуют
- для цели без шагов и без отметки выполнения отображается прогресс `0%`
Бизнес-правила
- пользователь не задает процент прогресса вручную
- если шаги отсутствуют, прогресс цели дискретный: `0%` или `100%`

История
ID: US-02
Название: Создание цели с шагами
Источник
- Пользовательский сценарий: пользователь создает цель с шагами и дальше отмечает выполненные шаги
- Функциональное требование: система должна позволять пользователю создать цель с шагами
- Функциональное требование: система должна автоматически рассчитывать прогресс цели по количеству выполненных шагов, если шаги заданы
Актор: Пользователь приложения по постановке личных целей
Потребность: создать цель с планом шагов
Ценность: отслеживать достижение цели через конкретные действия
Сценарий: пользователь создает цель и задает шаги, которые затем используются для вычисления прогресса
Критерии приемки
- при создании цели пользователь может добавить шаги
- сохраненные шаги становятся частью данных цели
- если у цели есть шаги, прогресс рассчитывается на основании выполненных шагов
- пользователь не может вручную ввести процент прогресса для цели со шагами
Бизнес-правила
- прогресс цели рассчитывается от шагов, если шаги были добавлены при создании цели
- процент прогресса недоступен для ручного редактирования пользователем

История
ID: US-03
Название: Отметка выполнения шагов и автоматический пересчет прогресса
Источник
- Пользовательский сценарий: пользователь создает цель с шагами и дальше отмечает выполненные шаги
- Функциональное требование: система должна позволять пользователю отмечать шаги цели как выполненные
- Функциональное требование: система должна автоматически переводить цель из `черновик` в `работаем` при выполнении первого шага
Актор: Пользователь приложения по постановке личных целей
Потребность: отмечать выполненные шаги
Ценность: видеть объективный прогресс по цели без ручных расчетов
Сценарий: пользователь открывает цель, отмечает выполненный шаг и получает обновленный прогресс и статус
Критерии приемки
- пользователь может отметить шаг как выполненный
- после отметки шага прогресс цели пересчитывается автоматически
- при выполнении первого шага цель из `черновик` переходит в `работаем`
- состояние выполненных шагов и новый процент прогресса сохраняются при повторном открытии цели
Бизнес-правила
- первый выполненный шаг автоматически переводит цель из `черновик` в `работаем`
- при наличии шагов прогресс рассчитывается только на основании шагов
Зависимости
- US-02

ID: US-04
Название: Ручное управление рабочим статусом цели
Источник
- Пользовательский сценарий: пользователь вручную переводит цель из `черновик` в `работаем`
- Пользовательский сценарий: пользователь вручную ставит цель на `паузу` и затем возобновляет работу
- Функциональное требование: система должна позволять пользователю вручную переводить цель из `черновик` в `работаем`
- Функциональное требование: система должна позволять пользователю вручную переводить цель из `работаем` в `пауза`
- Функциональное требование: система должна позволять пользователю вручную переводить цель из `пауза` в `работаем`
Актор: Пользователь приложения по постановке личных целей
Потребность: управлять рабочим состоянием цели вручную
Ценность: отражать реальный режим работы по цели, даже если шаги еще не выполнялись или работа временно остановлена
Сценарий: пользователь вручную меняет статус цели между `черновик`, `работаем` и `пауза` в допустимых пределах
Критерии приемки
- пользователь может вручную перевести цель из `черновик` в `работаем`
- пользователь может вручную перевести цель из `работаем` в `пауза`
- пользователь может вручную перевести цель из `пауза` в `работаем`
- пользователь не может вернуть цель в `черновик` после выхода из него
Бизнес-правила
- статус `пауза` устанавливается только вручную пользователем
- возврат цели в `черновик` не допускается

История
ID: US-05
Название: Завершение цели со шагами
Источник
- Пользовательский сценарий: пользователь завершает последний шаг и подтверждает завершение цели
- Альтернативный сценарий: пользователь завершает последний шаг, но отказывается завершать цель и добавляет новые шаги
- Функциональное требование: система должна предлагать пользователю завершить цель после выполнения последнего шага
- Функциональное требование: система должна автоматически переводить цель в `завершена` после подтверждения пользователем при выполненных всех шагах
- Функциональное требование: система должна позволять пользователю добавить новые шаги, если он отказался завершить цель после выполнения последнего шага
Актор: Пользователь приложения по постановке личных целей
Потребность: корректно завершать цель или продолжать ее, если после последнего шага появилась новая работа
Ценность: фиксировать реальное достижение цели без ложного завершения
Сценарий: пользователь завершает последний шаг, подтверждает завершение цели либо отказывается и добавляет новые шаги
Критерии приемки
- после выполнения последнего шага система предлагает завершить цель
- если пользователь подтверждает завершение, цель получает статус `завершена`
- если пользователь отказывается завершать цель, он может добавить новые шаги
- после добавления новых шагов прогресс цели пересчитывается
- завершенную цель нельзя вернуть в рабочие статусы
Бизнес-правила
- цель со шагами завершается автоматически после выполнения всех шагов и подтверждения пользователя
- если пользователь отказался завершить цель после выполнения последнего шага, он должен иметь возможность добавить новые шаги, а прогресс цели должен быть пересчитан
- статус `завершена` терминальный для сущности цели
Зависимости
- US-02
- US-03

История
ID: US-06
Название: Завершение цели без шагов и сохранение в достижениях
Источник
- Пользовательский сценарий: пользователь создает цель без шагов и отмечает ее выполненной целиком
- Пользовательский сценарий: пользователь открывает раздел достижений и видит завершенную цель
- Функциональное требование: система должна позволять пользователю явно отметить выполненной цель без шагов
- Функциональное требование: система должна показывать завершенные цели в разделе достижений
Актор: Пользователь приложения по постановке личных целей
Потребность: завершить цель без шагов и сохранить ее как достижение
Ценность: отличать достигнутую цель от отказа от нее и видеть результат в отдельном разделе
Сценарий: пользователь без шагов отмечает цель выполненной, после чего цель попадает в достижения
Критерии приемки
- для цели без шагов пользователь может явно отметить ее выполненной
- после отметки выполнения прогресс цели становится `100%`
- после завершения цель отображается в разделе достижений
- завершенную цель нельзя открыть снова для продолжения работы
- удаление цели не помещает ее в достижения
Бизнес-правила
- цель без шагов завершается после явной отметки выполнения пользователем
- завершенная цель должна отображаться в разделе достижений
- удаление цели означает отказ от цели, а завершение означает достижение
