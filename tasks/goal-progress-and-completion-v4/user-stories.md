# Пользовательские истории

Статус: завершено

История
ID: US-01
Название: Просмотр статуса и прогресса активной цели
Источник
- Пользовательский сценарий: пользователь открывает список активных целей и видит статус и прогресс каждой цели
- Функциональное требование: система должна показывать статус цели в списке активных целей и в карточке цели
- Функциональное требование: система должна показывать прогресс цели в процентах
Актор: Пользователь приложения по постановке личных целей
Потребность: видеть текущий статус и прогресс цели
Ценность: понимать, на каком этапе находится цель и есть ли по ней реальное продвижение
Сценарий: пользователь открывает список активных целей или карточку цели и видит актуальный статус и вычисленный процент прогресса
Критерии приемки
- в списке активных целей отображается текущий статус каждой цели
- в списке активных целей отображается текущий процент прогресса каждой цели
- в карточке цели отображаются статус, процент прогресса и состояние шагов, если они существуют
- для цели без шагов и без отметки выполнения отображается прогресс `0%`
Бизнес-правила
- пользователь не может вручную менять процент прогресса
- если у цели нет шагов, прогресс дискретный: `0%` или `100%`

История
ID: US-02
Название: Создание цели со шагами
Источник
- Пользовательский сценарий: пользователь создает цель со шагами и отмечает шаги выполненными
- Функциональное требование: система должна позволять пользователю создавать цель со шагами
- Функциональное требование: система должна автоматически рассчитывать прогресс как отношение выполненных шагов к общему количеству шагов, если шаги заданы
Актор: Пользователь приложения по постановке личных целей
Потребность: создать цель с планом шагов
Ценность: отслеживать достижение цели через конкретные действия
Сценарий: пользователь создает цель и добавляет к ней шаги, которые затем используются для расчета прогресса
Критерии приемки
- при создании цели пользователь может добавить шаги
- сохраненные шаги становятся частью данных цели
- если у цели есть шаги, прогресс рассчитывается на основании выполненных шагов
- пользователь не может вручную ввести процент прогресса для цели со шагами
Бизнес-правила
- прогресс цели со шагами рассчитывается только от выполненных шагов
- процент прогресса недоступен для ручного редактирования

История
ID: US-03
Название: Выполнение шагов и автоматический пересчет прогресса
Источник
- Пользовательский сценарий: пользователь создает цель со шагами и отмечает шаги выполненными
- Пользовательский сценарий: пользователь выполняет первый шаг и цель автоматически переходит из `план` в `в процессе`
- Функциональное требование: система должна позволять пользователю отмечать шаги выполненными
- Функциональное требование: система должна автоматически пересчитывать прогресс после изменения состояния шага
Актор: Пользователь приложения по постановке личных целей
Потребность: отмечать выполненные шаги
Ценность: видеть объективный прогресс по цели без ручных расчетов
Сценарий: пользователь открывает цель, отмечает шаг выполненным и получает обновленный прогресс и актуальный статус
Критерии приемки
- пользователь может отметить шаг как выполненный
- после отметки шага прогресс цели пересчитывается автоматически
- при выполнении первого шага цель из `план` переходит в `в процессе`
- состояние выполненных шагов и новый процент прогресса сохраняются при повторном открытии цели
Бизнес-правила
- выполнение первого шага автоматически переводит цель из `план` в `в процессе`
- при наличии шагов прогресс рассчитывается только на основании шагов
Зависимости
- US-02

История
ID: US-04
Название: Ручное управление рабочими статусами цели
Источник
- Пользовательский сценарий: пользователь вручную ставит цель на `паузу` и затем возвращает ее в `в процессе`
- Альтернативный сценарий: пользователь вручную переводит цель из `план` в `в процессе`, не дожидаясь выполнения первого шага
- Функциональное требование: система должна позволять пользователю вручную переводить цель между статусами `план`, `в процессе` и `пауза`
Актор: Пользователь приложения по постановке личных целей
Потребность: явно управлять рабочим состоянием цели
Ценность: отражать фактический режим работы по цели, даже если шаги еще не выполнялись или работа временно остановлена
Сценарий: пользователь вручную меняет статус цели между `план`, `в процессе` и `пауза` в допустимых пределах
Критерии приемки
- пользователь может вручную перевести цель из `план` в `в процессе`
- пользователь может вручную перевести цель из `в процессе` в `пауза`
- пользователь может вручную перевести цель из `пауза` в `в процессе`
- пользователь не может вручную установить статус `завершена`, пока не выполнены условия завершения
Бизнес-правила
- статусы `план`, `в процессе`, `пауза` могут меняться вручную
- статус `завершена` не устанавливается вручную для цели со шагами

История
ID: US-05
Название: Завершение цели со шагами
Источник
- Пользовательский сценарий: пользователь завершает цель со шагами после выполнения всех шагов
- Функциональное требование: система должна позволять завершить цель со шагами только после выполнения всех шагов
- Функциональное требование: система должна отличать завершение цели от удаления цели
Актор: Пользователь приложения по постановке личных целей
Потребность: завершить цель после выполнения всех запланированных шагов
Ценность: зафиксировать достижение как результат, а не терять его через удаление
Сценарий: пользователь выполняет все шаги цели, после чего цель получает статус `завершена` и больше не считается активной
Критерии приемки
- цель со шагами нельзя завершить, пока не выполнены все шаги
- после выполнения всех шагов цель получает статус `завершена`
- завершенная цель исчезает из списка активных целей
- завершенная цель отображается в достижениях
- завершенную цель нельзя вернуть в рабочие статусы
Бизнес-правила
- статус `завершена` достигается только при выполнении всех шагов
- завершение цели означает достижение, а не удаление
Зависимости

История
ID: US-06
Название: Завершение цели без шагов и показ в достижениях
Источник
- Пользовательский сценарий: пользователь завершает цель без шагов явной отметкой выполнения
- Пользовательский сценарий: пользователь открывает достижения и видит там завершенные цели
- Функциональное требование: система должна позволять пользователю явно отметить выполненной цель без шагов
- Функциональное требование: система должна переносить завершенную цель в достижения
Актор: Пользователь приложения по постановке личных целей
Потребность: завершить цель без шагов и сохранить ее как достижение
Ценность: отличать достигнутую цель от отказа от нее и видеть результат в отдельном разделе
Сценарий: пользователь у цели без шагов явно отмечает выполнение, после чего цель завершается и переносится в достижения
Критерии приемки
- для цели без шагов пользователь может явно отметить ее выполненной
- после отметки выполнения прогресс цели становится `100%`
- после завершения цель отображается в достижениях
- после завершения цель не отображается в списке активных целей
- удаление цели не помещает ее в достижения
Бизнес-правила
- цель без шагов завершается после явной отметки выполнения пользователем
- завершенная цель должна отображаться в достижениях, а не в активном списке
- удаление цели означает отказ от цели, а завершение означает достижение
