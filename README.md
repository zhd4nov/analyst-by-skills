# analyst-by-skills

Система для работы с требованиями через оркестратор и набор специализированных скиллов.

Оркестратор сам не пишет спецификацию, истории или gap-анализ. Он:
- принимает вход
- выбирает следующий этап
- передает контекст между этапами
- инициирует сохранение канонических артефактов через служебного агента

Для файловой фиксации артефактов используется служебный агент `artifact-persistence-agent`.
Он не анализирует требования и не меняет содержание артефактов, а только управляет каталогом прогона, именами файлов, версиями и пометками незавершенности.

Подробные правила находятся в [AGENTS.md](./AGENTS.md).

## Что делает система

Система принимает входной материал и проводит его через пайплайн требований.

На выходе получаются:
1. `Спецификация`
2. `Пользовательские истории`
3. `Отчет о пробелах и рисках`

Дополнительно система может сохранять:
- `Уточненные требования`
- `Допущения`
- `Открытые вопросы`
- `Статус готовности историй`

## Как работает пайплайн

Основные этапы:
1. `requirements-elicitor`
2. `spec-structurer`
3. `story-extractor`
4. `story-quality-reviewer`
5. `requirements-gap-analyzer`

Оркестратор запускает не все этапы подряд автоматически. Он выбирает следующий шаг по состоянию артефактов.

Примеры:
- если вход сырой или неоднозначный, сначала вызывается `requirements-elicitor`
- если вход уже близок к спецификации, система может начать с `spec-structurer`
- если на любом этапе появились новые критичные `Открытые вопросы`, пайплайн не идет дальше, пока они не закрыты

## Какие входы подходят

Система рассчитана на такие типы входа:
- сырой текст с идеей или проблемой
- заметки встречи
- переписка
- частичная спецификация
- набор пользовательских историй
- смешанный ввод

Любой предметный материал, который пользователь передает для анализа, считается артефактом `Вход`.

## Что сохраняется

Каждый прогон сохраняется в `tasks/<name-of-task>/`.

Минимальный набор файлов:
- `input.md`
- `clarification-log.md`
- `clarified-requirements.md`
- `canonical-rules.md`
- `assumptions.md`
- `open-questions.md`
- `specification.md`
- `user-stories.md`
- `story-readiness.md`
- `gap-risk-report.md`

Если каталог задачи уже существует, `artifact-persistence-agent` должен создать новый каталог-версию рядом. Существующие каталоги в `tasks/` не переписываются.

## Как использовать систему

Обычный режим работы:
1. Передайте входной материал с требованиями.
2. Сформулируйте, что нужен анализ требований, спецификация, истории или gap-анализ.
3. Ответьте на уточняющие вопросы, если система остановила пайплайн.
4. Заберите итоговые артефакты из ответа и из каталога `tasks/...`.

Типичные запросы:
- "Разбери это как требования"
- "Сделай спецификацию и истории"
- "Проведи gap-анализ"
- "Вот заметки встречи, запусти пайплайн"

## Структура репозитория

```text
.
├── AGENTS.md
├── README.md
├── examples/
├── agents/
├── scenarios/
├── templates/
├── skills/
└── tasks/
```

Ключевые каталоги:
- `agents/` — служебные агенты оркестратора
- `skills/` — скиллы этапов пайплайна
- `templates/` — минимальные шаблоны артефактов
- `scenarios/` — сценарии маршрутизации
- `examples/` — примеры полного прохода
- `tasks/` — сохраненные результаты прогонов

## Режимы работы

Система работает в двух режимах:
- `requirements-processing`
- `system-maintenance`

По умолчанию, если пользователь приносит требования или просит запустить анализ, используется `requirements-processing`.

## Режим обслуживания

Чтобы перейти в режим обслуживания, нужно отправить команду `РЕМОНТ`.

После этого система:
- не запускает продуктовый пайплайн автоматически
- трактует запросы как работу над самой системой
- редактирует контракты, шаблоны, скиллы, примеры и правила оркестрации

Этот режим нужен, когда требуется:
- изменить `AGENTS.md`
- упростить или ужесточить правила навыков
- поправить шаблоны
- обновить примеры и сценарии
- проверить, соответствует ли документация текущей реализации
