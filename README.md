# analyst-by-skills

Репозиторий описывает систему агентной обработки требований, построенную как оркестратор и набор специализированных скиллов.

Главная идея проекта: не пытаться "сразу написать решение", а последовательно проходить путь от сырого входа к проверяемым артефактам требований. Система исходит из того, что качество спецификации, пользовательских историй и анализа рисков определяется качеством отправной точки. Поэтому в центре проекта стоит оркестратор, который запускает нужные скиллы по шагам, сохраняет контекст между этапами и не подменяет собой специализированный анализ.

## Что делает система

Система преобразует входные материалы в три итоговых артефакта:

1. Спецификация
2. Пользовательские истории
3. Отчет о пробелах и рисках

Поддерживаются разные типы входа:
- сырой ввод
- заметки встречи
- переписка
- псевдо-спецификация
- отдельная пользовательская история
- смешанный ввод

## Архитектура

В проекте есть два уровня логики:

- [AGENTS.md](./AGENTS.md) задает правила оркестрации, маршрутизацию, канонические артефакты, финальные проверки качества и правила повторного вызова этапов.
- Каталог `skills/` содержит специализированные скиллы, каждый из которых отвечает только за свой участок пайплайна.
- Каталог `templates/` содержит канонические минимально достаточные шаблоны артефактов.
- Каталог `scenarios/` содержит проверочные сценарии маршрутизации оркестратора.
- Каталог `examples/` содержит сквозные примеры полного прохода через систему.

Текущие скиллы:

- `requirements-elicitor`
- `spec-structurer`
- `story-extractor`
- `story-quality-reviewer`
- `requirements-gap-analyzer`

## Пайплайн

Оркестратор работает по последовательному сценарию:

1. Классификация входа
2. Уточнение требований
3. Структурирование спецификации
4. Извлечение пользовательских историй
5. Проверка качества историй
6. Анализ пробелов и рисков
7. Агрегация итогового результата

На каждом этапе сохраняются:
- допущения
- открытые вопросы
- контекст, важный для следующего шага

Если новые `Открытые вопросы` впервые появились на этапе структурирования спецификации, система не должна просто нести их дальше. Оркестратор обязан сразу попытаться их закрыть через повторное уточнение и только потом продолжать пайплайн.

## Принципы

Проект опирается на несколько базовых принципов:

- требования важнее преждевременных решений
- корректность важнее полноты
- ясность важнее объема
- проверяемость важнее формальности
- если информации не хватает, это нужно фиксировать явно, а не заполнять догадками

## Текущее состояние

Сейчас репозиторий содержит:

- корневой файл оркестрации
- пять базовых скиллов
- контракты входов и выходов для каждого скилла
- сценарии сбоев и правила повторного вызова
- примеры для каждого скилла, показывающие важность сильной отправной точки и требований как фундамента
- шаблоны артефактов в простом текстовом формате
- минимальные правила трассируемости между спецификацией, историями и пробелами
- сценарии маршрутизации оркестратора
- один сквозной пример полного прохода от входа до итоговых артефактов

Следующий естественный этап развития проекта:
- при необходимости расширить набор сквозных примеров
- при необходимости добавить дополнительные доменные сценарии
- при необходимости автоматизировать проверки на уровне кода или runner-логики
