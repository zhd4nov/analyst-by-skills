# Шаблон: Пользовательские истории

Использовать как минимально достаточный результат после этапа извлечения историй или проверки их качества.

```text
Пользовательские истории

История
ID: <идентификатор>
Название: <краткое название истории>
Источник
- <пользовательский сценарий / функциональное требование / бизнес-правило>
- <дополнительный источник, если нужен для трассируемости>
Актор: <кто получает ценность>
Потребность: <что нужно сделать>
Ценность: <зачем это нужно актору или бизнесу>
Сценарий: <краткий основной сценарий>
Критерии приемки
- <критерий 1>
- <критерий 2>
Бизнес-правила
- <правило 1>
- <правило 2>
Зависимости
- <другая история / внешняя система / решение по открытому вопросу>
- <другая история / внешняя система / решение по открытому вопросу>
Открытые вопросы
- <вопрос 1>
- <вопрос 2>

Название: <краткое название истории>
Источник
- <пользовательский сценарий / функциональное требование / бизнес-правило>
- <дополнительный источник, если нужен для трассируемости>
Актор: <кто получает ценность>
Потребность: <что нужно сделать>
Ценность: <зачем это нужно актору или бизнесу>
Сценарий: <краткий основной сценарий>

- <критерий 2>

```

## Правила использования

- Одна история должна описывать одну самостоятельную ценность.
- Если история покрывает несколько разных сценариев, ее нужно разделить.
- `Ценность` должна быть явной, а не подразумеваемой.
- Критерии приемки должны быть проверяемыми и конкретными.
- `Источник` должен указывать, из чего именно в спецификации появилась история. Достаточно одного явного источника, но при наличии связанного сценария и правила их нужно сохранять оба.
- `Зависимости` должны быть конкретными. Если зависимостей нет, нужно писать `нет`.
- `Открытые вопросы` должны содержать только то, что влияет на полноту или готовность истории. Если вопросов нет, нужно писать `нет`.
