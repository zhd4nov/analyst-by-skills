# Шаблон: Уточненные требования

Использовать как минимально достаточный результат после этапа уточнения требований.

```text
Уточненные требования

Акторы
- <основной актор>
- <дополнительный актор>

Цели
- <какую бизнес- или пользовательскую цель нужно достичь>

Ключевые сценарии
- <основной сценарий 1>
- <основной сценарий 2>

Известные ограничения
- <ограничение 1>
- <ограничение 2>

Допущения
- <подтвержденная пользователем рабочая гипотеза 1>
- <подтвержденная пользователем рабочая гипотеза 2>

Открытые вопросы
- <блокер / важный>: <вопрос без ответа>
  - Почему открыт: <какой информации не хватает или почему ответ не удалось получить на этапе уточнения>
  - Где уточнить: <у собеседника / у владельца процесса / у разработчика / в регламенте / в другой системе>
```

## Правила использования

- Не добавлять разделы, если для них нет содержания.
- Не превращать допущения в факты.
- В `Допущения` должны попадать только рабочие гипотезы, явно подтвержденные пользователем на этапе уточнения.
- Если цели или сценарии не определены, это нужно явно показать в `Открытые вопросы`.
- В `Открытые вопросы` должны попадать только вопросы, которые не удалось закрыть на текущем этапе уточнения.
- Если в ходе уточнения вопрос признан несущественным для текущего функционального объема, он не должен попадать в `Открытые вопросы`.
- Если пользователь отверг рабочую гипотезу, она не должна попадать в `Допущения`.
- У каждого открытого вопроса должно быть краткое объяснение, почему он остался открытым, и где искать ответ.
- Формулировки должны описывать проблему и намерение, а не интерфейсное решение.
