# Сквозной Пример: Статус Возврата

Использовать этот файл как канонический минимальный пример полного прохода через систему.

---

## Вход

- "Клиенты часто спрашивают, где их возврат. Хотим, чтобы они сами видели статус возврата в личном кабинете. На складе статус меняют вручную. Частичный возврат пока не нужен."

---

## Уточненные требования

Акторы
- Клиент
- Оператор склада

Цели
- Клиент должен понимать текущее состояние возврата без обращения в поддержку
- Компания должна сократить число типовых обращений по возвратам

Ключевые сценарии
- клиент открывает возврат и видит текущий статус
- оператор склада меняет статус возврата после обработки

Известные ограничения
- статус обновляется на основании данных склада
- частичный возврат не входит в текущий объем

Допущения
- каждый возврат имеет один текущий статус
- история смены статусов не обязательна для первой версии

Открытые вопросы
- важный: как быстро статус со склада должен появляться у клиента?
- важный: должен ли клиент видеть причину отклонения возврата?

---

## Спецификация

Контекст / Проблема
Клиенты не понимают, на каком этапе находится возврат, и обращаются в поддержку за статусом вручную.

Бизнес-цель
Снизить число обращений в поддержку по вопросам статуса возврата.

Границы объема
- Внутри объема:
  - отображение текущего статуса возврата клиенту
  - изменение статуса возврата оператором склада
- Вне объема:
  - частичный возврат
  - расчет суммы возврата

Акторы
- Клиент
- Оператор склада

Пользовательские сценарии
- клиент открывает возврат и видит текущий статус
- оператор склада меняет статус возврата после приемки товара

Функциональные требования
- система отображает клиенту текущий статус возврата
- система позволяет оператору склада изменять статус возврата

Бизнес-правила
- возврат относится к конкретному заказу
- частичный возврат не поддерживается в текущей версии

Данные / Сущности
- Заказ
- Возврат
- Статус возврата

Ограничения
- источник статуса находится на стороне склада

Допущения
- клиенту достаточно видеть текущий статус без полной истории изменений

Открытые вопросы
- как быстро статус должен синхронизироваться?
- должен ли клиент видеть причину отклонения?

Зависимости

- Функциональное требование: система позволяет оператору склада изменять статус возврата
Актор: Оператор склада
Потребность: менять статус возврата после обработки
Ценность: поддерживать для клиента актуальную информацию
Сценарий: оператор склада меняет статус после приемки товара
Критерии приемки
- оператор может выбрать новый статус из допустимого набора
- новый статус сохраняется в системе
- новый статус становится доступен клиенту
Бизнес-правила
- частичный возврат не поддерживается
Зависимости
- нет

## Отчет о пробелах и рисках

Пробел
ID: GAP-01
Источник пробела
- Спецификация -> Ограничения
- Открытые вопросы -> как быстро статус должен синхронизироваться?
Область: Интеграция
Описание пробела: не определена допустимая задержка обновления статуса возврата
Почему это важно: клиент может видеть устаревший статус и не доверять системе
Риск: средний
Влияние: сохранится часть обращений в поддержку, несмотря на новую функцию
Что нужно уточнить: какой интервал обновления считается приемлемым для бизнеса и клиента
Предлагаемое решение: определить целевое время обновления статуса
Блокирует разработку: нет

Пробел
ID: GAP-02
Источник пробела
- Пользовательские истории -> US-01
- Спецификация -> Открытые вопросы
Область: Отображение статуса
Описание пробела: не определено, должен ли клиент видеть причину отклонения возврата
Почему это важно: без этого нельзя корректно определить состав отображаемых данных
Риск: средний
Влияние: возможна доработка интерфейса и логики уже после реализации
Что нужно уточнить: нужно ли показывать причину отклонения и в каком виде
Предлагаемое решение: согласовать правило отображения причины отклонения
Блокирует разработку: нет

## Что показывает этот пример

- Система должна начинать с проблемы и цели, а не с экрана.
- Спецификация должна создавать фундамент для историй.
- Истории должны иметь явное происхождение.
- Пробелы должны ссылаться на конкретные места, а не формулироваться абстрактно.
- Даже в хорошем потоке часть вопросов может остаться открытой, и это нормально для MVP.
