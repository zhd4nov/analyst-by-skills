# Пользовательские истории

## История
ID: US-01
Название: Подача заявки на командировку
Источник
- `Спецификация -> Пользовательские сценарии -> Сотрудник создает и отправляет заявку на командировку`
- `Спецификация -> Функциональные требования -> создать заявку с датами, городом, целью поездки и предварительным бюджетом`
- `CR-01`
- `CR-02`
Актор: сотрудник
Потребность: создать и отправить заявку на командировку с ключевыми параметрами поездки.
Ценность: запустить согласование командировки без переписок в почте и мессенджерах.
Сценарий: сотрудник заполняет даты, город, цель поездки и предварительный бюджет, затем отправляет заявку руководителю.
Критерии приемки
- Заявка не может быть отправлена без дат, города, цели поездки и предварительного бюджета.
- После отправки заявка получает статус `На согласовании у руководителя`.
- После отправки заявка доступна руководителю для решения.
Бизнес-правила
- После отправки заявка должна попасть руководителю на согласование.
Зависимости
- нет

## История
ID: US-02
Название: Согласование заявки руководителем
Источник
- `Спецификация -> Пользовательские сценарии -> Руководитель согласует отправленную заявку`
- `Спецификация -> Модель состояний / Переходы -> На согласовании у руководителя -> Согласована руководителем`
- `CR-03`
Актор: руководитель
Потребность: согласовать заявку на командировку.
Ценность: зафиксировать управленческое решение по поездке в едином процессе.
Сценарий: руководитель открывает заявку, находящуюся на согласовании, и согласует ее.
Критерии приемки
- Руководитель может согласовать заявку, находящуюся на согласовании.
- После согласования заявка получает статус `Согласована руководителем`.
- После согласования заявка становится доступна бухгалтерии.
Бизнес-правила
- Согласованная руководителем заявка передается в зону видимости бухгалтерии.
Зависимости
- Наличие отправленной заявки.
Открытые вопросы
- нет

## История
ID: US-03
Название: Отклонение заявки руководителем
Источник
- `Спецификация -> Пользовательские сценарии -> Руководитель отклоняет заявку и указывает причину отклонения`
- `Спецификация -> Валидационные правила -> Отклонение заявки руководителем без причины недопустимо`
- `CR-04`
Актор: руководитель
Потребность: отклонить заявку с причиной.
Ценность: зафиксировать понятное решение и дать сотруднику объяснение отказа.
Сценарий: руководитель открывает заявку на согласовании, указывает причину и отклоняет заявку.
Критерии приемки
- Руководитель не может отклонить заявку без причины.
- После отклонения заявка получает статус `Отклонена руководителем`.
- Причина отклонения сохраняется в заявке.
Бизнес-правила
- Причина отклонения обязательна.
Зависимости
- Наличие отправленной заявки.
Открытые вопросы
- нет

## История
ID: US-04
Название: Просмотр статуса заявки сотрудником
Источник
- `Спецификация -> Функциональные требования -> показывать сотруднику статус его заявки`
- `Спецификация -> Данные / Сущности -> статус`
- `CR-05`
Актор: сотрудник
Потребность: видеть статус своей заявки на командировку.
Ценность: понимать, согласована ли поездка и что происходит с подготовкой документов.
Сценарий: сотрудник открывает свою заявку и видит текущий статус.
Критерии приемки
- Сотрудник видит текущий статус своей заявки.
- Статус отражает одно из состояний, зафиксированных в спецификации.
- После действий руководителя или бухгалтерии сотрудник видит обновленный статус заявки.
Бизнес-правила
- Статус заявки должен отражать фактический этап жизненного цикла заявки.
Зависимости
- Наличие созданной заявки.

## История
ID: US-05
Название: Подготовка документов бухгалтерией
Источник
- `Спецификация -> Пользовательские сценарии -> Бухгалтерия видит согласованную заявку и отмечает завершение подготовки аванса и документов`
- `Спецификация -> Бизнес-правила -> Бухгалтерия не согласует поездку`
- `CR-06`
- `CR-07`
- `CR-08`
Актор: бухгалтерия
Потребность: видеть согласованные заявки и отмечать завершение подготовки аванса и документов.
Ценность: выполнять финансовую и документальную подготовку без дополнительных уточнений в переписке.
Сценарий: бухгалтерия открывает согласованную заявку, подготавливает аванс и документы, затем отмечает подготовку завершенной.
Критерии приемки
- Бухгалтерия видит заявку после согласования руководителем.
- Бухгалтерия может отметить завершение подготовки аванса и документов.
- Бухгалтерия не может согласовать или отклонить поездку вместо руководителя.
- После отметки бухгалтерии заявка получает статус `Документы подготовлены`.
Бизнес-правила
- Бухгалтерия не согласует поездку.
- Бухгалтерия работает только с согласованными заявками.
Зависимости
- Заявка согласована руководителем.
Открытые вопросы

## История
ID: US-06
Название: Изменение отправленной заявки с повторным согласованием
Источник
- `Спецификация -> Альтернативные сценарии -> Сотрудник меняет даты или бюджет после отправки`
- `Спецификация -> Валидационные правила -> изменение дат или бюджета после отправки возвращает заявку на повторное согласование`
- `Спецификация -> Бизнес-правила -> до повторного согласования обновленная заявка не считается утвержденной`
- `Лог уточнений: CL-01`
- `CR-09`
- `CR-10`
- `CR-11`
Актор: сотрудник
Потребность: изменить даты или бюджет в уже отправленной заявке.
Ценность: исправить существенные параметры поездки без скрытого изменения ранее согласованной заявки.
Сценарий: сотрудник изменяет даты или бюджет после отправки заявки, после чего заявка возвращается руководителю на повторное согласование.
Критерии приемки
- Если сотрудник меняет даты после отправки, заявка получает статус `На повторном согласовании у руководителя`.
- Если сотрудник меняет бюджет после отправки, заявка получает статус `На повторном согласовании у руководителя`.
- До повторного согласования обновленная заявка не считается утвержденной.
- Руководитель может повторно согласовать или отклонить обновленную заявку с причиной отклонения.
Бизнес-правила
- После отправки сотрудник не может незаметно изменить заявку.
- Изменение дат или бюджета после отправки требует повторного согласования руководителем.
- До повторного согласования обновленная заявка не считается утвержденной.
Зависимости
- Наличие отправленной заявки.
