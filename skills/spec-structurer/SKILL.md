# spec-structurer

## Назначение
Преобразует текст в минимально-достаточную спецификацию.

## Вход
- сырой текст или уточнённые требования

## Выход
- спецификация

## Контракт входа
- `Вход`
- `Уточненные требования` (опционально)
- `Допущения`
- `Открытые вопросы`

## Контракт выхода
- `Спецификация`

## Сценарии сбоя
- отсутствуют обязательные разделы
- в спецификации есть явные противоречия
- допущения потеряны или смешаны с требованиями
- открытые вопросы исчезли после структурирования
- границы объема сформулированы неявно

## Правила повторного вызова
Повторный вызов нужен, если:
- отсутствует хотя бы один обязательный раздел
- есть конфликт между разделами
- допущения и открытые вопросы не сохранены явно

При повторном вызове нужно передавать:
- текущую `Спецификацию`
- исходный `Вход`
- `Уточненные требования` (если есть)
- список конкретных проблем

## Правила

1. Удалять дубли и “воду”
2. Не придумывать отсутствующие требования
3. Явно помечать допущения

## Формат

- Контекст / Проблема
- Бизнес-цель
- Границы объема (внутри / вне объема)
- Акторы
- Пользовательские сценарии
- Функциональные требования
- Бизнес-правила
- Данные / Сущности
- Ограничения
- Допущения
- Открытые вопросы

## Примеры

### Пример 1: Из уточнений в рабочую спецификацию

#### Вход
- `Уточненные требования`:
  - Акторы: Клиент, Оператор склада
  - Цели: Клиент должен видеть статус возврата товара без звонка в поддержку
  - Ключевые сценарии: создание заявки на возврат, прием возврата складом, изменение статуса возврата
  - Известные ограничения: статусы берутся из существующей складской системы
- `Допущения`:
  - возврат создается только по оплаченному заказу
- `Открытые вопросы`:
  - нужен ли частичный возврат?

#### Ожидаемый выход
- `Спецификация`:
  - Контекст / Проблема: клиенты не понимают, на каком этапе находится возврат
  - Бизнес-цель: снизить число обращений в поддержку по статусу возврата
  - Границы объема:
    - внутри объема: просмотр статуса, история смены статусов
    - вне объема: расчет суммы возврата
  - Акторы: Клиент, Оператор склада
  - Пользовательские сценарии: клиент открывает возврат и видит актуальный статус
  - Функциональные требования: система отображает текущий статус возврата
  - Бизнес-правила: возврат доступен только для оплаченного заказа
  - Данные / Сущности: заказ, возврат, статус возврата
  - Ограничения: источник статусов внешняя складская система
  - Допущения: частичный возврат пока не рассматривается
  - Открытые вопросы: нужен ли частичный возврат?

#### Что показывает пример
- Хорошая спецификация начинается не с набора функций, а с контекста и бизнес-цели.
- Требования должны собираться на фундаменте проблемы, а не на списке хотелок.

### Пример 2: Нельзя перепрыгивать через смысл к фичам

#### Вход
- "Нужны фильтры, таблица, цветовые статусы и экспорт в Excel для возвратов."

#### Ожидаемый выход
- `Спецификация`:
  - Контекст / Проблема: не определен явно
  - Бизнес-цель: не определена явно
  - Границы объема:
    - внутри объема: отображение списка возвратов
    - вне объема: не определено
  - Акторы: не определены явно
  - Пользовательские сценарии: не определены явно
  - Функциональные требования: отображение списка, фильтрация, экспорт
  - Бизнес-правила: не определены
  - Данные / Сущности: возврат
  - Ограничения: не определены
  - Допущения: фильтры нужны для операционной обработки
  - Открытые вопросы: кто использует таблицу и для каких решений?

#### Что показывает пример
- Если на входе только набор элементов интерфейса, спецификатор должен зафиксировать пробелы, а не делать вид, что требования уже готовы.
- Сначала смысл, затем структура, затем решение.
