# requirements-elicitor

## Назначение
Преобразует сырой ввод в уточнённый набор требований.

## Когда использовать
- вход неполный
- отсутствуют акторы
- нет целей или сценариев

## Контракт входа
- `Вход`
- `Лог уточнений` (опционально)
- текущие `Уточненные требования` (при повторном вызове)
- текущие `Канонические правила` (при повторном вызове)
- новые `Открытые вопросы` от downstream-этапа (при возврате на уточнение)
- список конкретных пробелов или замечаний, которые нужно закрыть (при повторном вызове)

## Контракт выхода
- сигнал этапа: `complete`, `needs_user_answer` или `needs_quality_rework`
- `Лог уточнений`
- `Уточненные требования`
- `Канонические правила`
- `Допущения`
- `Открытые вопросы`

## Обязательная последовательность выполнения

1. Определить, хватает ли входа для фиксации:
   - акторов
   - целей
   - текущего функционального объема
   - ключевых сценариев
   - существенных бизнес-правил и ограничений
2. Если не хватает хотя бы одного существенного элемента:
   - не формировать завершенный результат этапа уточнения
   - вернуть сигнал `needs_user_answer`
   - вернуть короткий список уточняющих вопросов по приоритету
3. Вопросы задаются в порядке:
   - блокеры
   - важные
   - второстепенные
4. После каждого ответа пользователя:
   - обновить рабочее понимание требований
   - отделить подтвержденные факты от гипотез
   - добавить запись в `Лог уточнений`, если ответ изменил понимание требований
   - при необходимости вернуть следующий вопрос с сигналом `needs_user_answer`
5. `Открытые вопросы` формируются только для тех пунктов, которые:
   - уже пытались уточнить через диалог или были возвращены downstream-этапом для немедленного закрытия
   - не удалось закрыть из `Входа`, текущего диалога и полученного ответа пользователя
   - действительно существенны для текущего функционального объема
6. После фиксации итоговых фактов:
   - собрать `Канонические правила` как отдельный компактный слой
   - разложить подтвержденные факты на атомарные правила
   - для каждого правила указать источник и downstream-назначение

## Состояния этапа уточнения

- `needs_user_answer`: обнаружены существенные пробелы, и для продолжения нужен ответ пользователя
- `needs_quality_rework`: результат этапа нарушает контракт скилла и требует повторной генерации без нового ответа пользователя
- `complete`: блокирующие вопросы закрыты; оставшиеся `Открытые вопросы` не мешают следующему этапу либо пользователь явно подтвердил, что ответ сейчас недоступен

Скилл не должен возвращать `complete`, если обнаруженные существенные пробелы еще можно закрыть вопросом к пользователю и без ответа нельзя безопасно перейти к следующему этапу.

## Сценарии сбоя
- не выделены акторы
- не выделены цели
- не зафиксирован текущий функциональный объем, хотя он следует из входа и уточнения
- не выделены ключевые сценарии
- допущения скрыто смешаны с фактами
- в `Допущения` попала рабочая гипотеза без явного подтверждения пользователем
- открытые вопросы слишком общие и не помогают снять блокеры
- открытые вопросы попали в результат без попытки уточнения через доступный диалог
- у открытого вопроса нет объяснения, почему он остался открытым
- у открытого вопроса не указано, где или у кого его нужно уточнять
- в `Открытые вопросы` попал вопрос, признанный несущественным для текущего функционального объема
- факт получен из ответа пользователя, но не отражен в `Логе уточнений`
- `Лог уточнений` содержит вопрос без ответа, если ответ уже был получен
- `Канонические правила` отсутствуют, хотя этап завершен и подтвержденные факты уже есть
- одно правило в `Канонических правилах` смешивает несколько независимых утверждений
- у правила нет явного источника или непонятно, зачем оно нужно downstream-этапам

## Правила

1. Не додумывать факты
2. Если для движения нужна рабочая гипотеза, сформулировать ее как проверяемый вопрос пользователю в формате `да / нет`
3. В `Допущения` включать только те рабочие гипотезы, которые пользователь подтвердил
4. Если пользователь отверг гипотезу, можно предложить альтернативную, но не более трех гипотез по одной существенной теме
5. После трех отвергнутых гипотез прекратить угадывание: сохранить `Открытый вопрос`, если тема существенна, или не сохранять ничего, если тема несущественна
6. Сначала пытаться снять неоднозначность из доступного входа и текущего диалога, а если это невозможно, вернуть вопрос пользователю с сигналом `needs_user_answer`
7. Задавать вопросы по приоритету:
   - блокеры
   - важные
8. В `Открытые вопросы` переносить только то, что не удалось уточнить на текущем этапе после вопроса пользователю или после явного ответа пользователя, что информация пока неизвестна
9. Если в ходе уточнения вопрос признан несущественным для текущего функционального объема, не включать его в `Открытые вопросы`
10. Для каждого открытого вопроса указывать, почему он остался открытым и где искать ответ
11. Не удалять расплывчатые формулировки вроде “удобно” или “быстро” молча: если из источника есть наблюдаемый критерий, заменить формулировку на него; если критерия нет, вернуть вопрос пользователю
12. Не считать внутренний анализ заменой пользовательского уточнения
13. Не возвращать завершенный результат этапа, если существенные вопросы еще не были заданы пользователю
14. Если пользователь явно просит промежуточный результат до завершения уточнения, помечать такой результат как незавершенный
15. Если новый факт получен из ответа пользователя, он должен быть зафиксирован в `Логе уточнений` до включения в `Уточненные требования`
16. `Лог уточнений` должен содержать только вопросы и ответы, которые реально повлияли на понимание требований
17. `Уточненные требования` должны содержать только итоговые факты; текст вопроса и ответа хранится только в `Логе уточнений`
18. Если факт отсутствовал во `Входе`, но подтвержден ответом пользователя, рядом с фактом в `Уточненных требованиях` нужно указывать ссылку `[Лог уточнений: CL-xx]`
19. После завершения уточнения нужно собрать отдельные `Канонические правила` как список атомарных фактов для downstream-этапов
20. `Канонические правила` не должны заменять `Уточненные требования`: первые нужны для операционального использования, вторые для сохранения человеческого контекста
21. В `Канонические правила` включать только подтвержденные факты, которые влияют на поведение, валидацию, термины, переходы, ограничения, видимость или состав данных
22. У каждого правила в `Канонических правилах` должен быть явный источник и указание, где оно должно быть использовано дальше

## Запреты

Скилл не должен:
- возвращать завершенный результат этапа уточнения, если существенные вопросы еще не были заданы пользователю
- подменять обязательный диалог списком `Открытые вопросы`
- считать попытку внутреннего анализа эквивалентом пользовательского уточнения
- завершать этап только потому, что удалось сформулировать правдоподобный результат
- включать в `Уточненные требования` факт из ответа пользователя без записи в `Логе уточнений`
- дублировать в `Уточненных требованиях` текст вопроса или ответа, уже сохраненный в `Логе уточнений`
- смешивать несколько независимых фактов в одном каноническом правиле
- пропускать сборку `Канонических правил`, если есть материал для их формирования

## Самопроверка перед возвратом результата

Перед возвратом завершенного результата скилл обязан проверить:
- все ли существенные вопросы были заданы пользователю или исчерпаны допустимые попытки уточнения
- есть ли `Акторы`
- есть ли `Цели`
- есть ли `Текущий функциональный объем`
- есть ли `Ключевые сценарии`
- нет ли в `Допущения` неподтвержденных гипотез
- не попали ли в `Открытые вопросы` пункты, которые еще не пытались уточнить
- зафиксированы ли в `Логе уточнений` ответы, которые изменили требования
- собраны ли `Канонические правила` и покрывают ли они ключевые подтвержденные факты

Если хотя бы один ответ отрицательный, этап не считается завершенным.

## Результат этапа

- отдельный артефакт `Лог уточнений`
- `Уточненные требования` со структурой:
  - Акторы
  - Цели
  - Текущий функциональный объем
  - Ключевые сценарии
  - Известные ограничения
- отдельный артефакт `Канонические правила` со структурой:
  - ID правила
  - Тип
  - Формулировка
  - Источник
  - Использовать дальше
- `Допущения`
- `Открытые вопросы`
