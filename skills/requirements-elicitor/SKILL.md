# requirements-elicitor

## Назначение
Преобразует сырой ввод в уточнённый набор требований.

## Когда использовать
- вход неполный
- отсутствуют акторы
- нет целей или сценариев

## Вход
- сырой ввод

## Выход
- уточненные требования
- допущения
- открытые вопросы

## Контракт входа
- `Вход`

## Контракт выхода
- `Уточненные требования`
- `Допущения`
- `Открытые вопросы`

## Сценарии сбоя
- не выделены акторы
- не выделены цели
- не выделены ключевые сценарии
- допущения скрыто смешаны с фактами
- открытые вопросы слишком общие и не помогают снять блокеры
- открытые вопросы попали в результат без попытки уточнения через доступный диалог
- у открытого вопроса нет объяснения, почему он остался открытым
- у открытого вопроса не указано, где или у кого его нужно уточнять
- в `Открытые вопросы` попал вопрос, признанный несущественным для текущего функционального объема

## Правила повторного вызова
Повторный вызов нужен, если:
- отсутствует хотя бы один из разделов `Акторы`, `Цели`, `Ключевые сценарии`
- допущения не отделены явно
- открытые вопросы не отсортированы по приоритету
- открытые вопросы не объясняют, почему их не удалось закрыть на текущем этапе

При повторном вызове нужно передавать:
- исходный `Вход`
- текущие `Уточненные требования`
- доступный контекст уже состоявшегося уточнения
- список конкретных пробелов

## Правила

1. Не додумывать — только помечать допущения
2. Сначала пытаться снять неоднозначность из доступного входа и диалога, а не переносить ее в `Открытые вопросы`
3. Задавать вопросы по приоритету:
   - блокеры
   - важные
4. В `Открытые вопросы` переносить только то, что не удалось уточнить на текущем этапе
5. Если в ходе уточнения вопрос признан несущественным для текущего функционального объема, не включать его в `Открытые вопросы`
6. Для каждого открытого вопроса указывать, почему он остался открытым и где искать ответ
7. Удалять неценные формулировки (“удобно”, “быстро”)

## Результат (структура)

- Акторы
- Цели
- Ключевые сценарии
- Известные ограничения
- Допущения
- Открытые вопросы

## Примеры

### Пример 1: Начинать нужно с проблемы, а не с экрана

#### Вход
- "Нужен экран согласования заявок. Должно быть удобно для руководителя и быстро."

#### Ожидаемый выход
- `Уточненные требования`:
  - Акторы: Руководитель отдела, Инициатор заявки
  - Цели: Руководитель должен принимать решение по заявке без переписки в почте
  - Ключевые сценарии: просмотр новой заявки, согласование, отклонение с причиной
  - Известные ограничения: решение должно фиксироваться в системе
- `Допущения`:
  - причина отклонения обязательна
  - согласующий у заявки один
- `Открытые вопросы`:
  - блокер: кто может переназначать согласующего?
    - Почему открыт: во входе нет правила переназначения, и собеседник его не уточнил
    - Где уточнить: у владельца процесса согласования
  - важный: нужно ли повторное согласование после изменения заявки?
    - Почему открыт: вход описывает согласование, но не задает правило повторного маршрута после изменений
    - Где уточнить: у владельца процесса согласования

#### Что показывает пример
- Нельзя начинать с "экрана" как отправной точки.
- Сначала надо определить акторов, цель и сценарий, иначе интерфейс будет подменять собой требования.

### Пример 2: Плохая отправная точка дает слабые требования

#### Вход
- "Сделайте как в Jira, только проще. Нужно, чтобы менеджерам было удобно."

#### Ожидаемый выход
- `Уточненные требования`:
  - Акторы: Менеджер
  - Цели: не определены явно
  - Ключевые сценарии: не определены явно
  - Известные ограничения: не определены
- `Допущения`:
  - под "проще" может пониматься сокращение числа шагов
- `Открытые вопросы`:
  - блокер: какую именно задачу менеджер должен решать в системе?
    - Почему открыт: вход содержит только аналогию с внешним продуктом и не описывает целевую работу менеджера
    - Где уточнить: у собеседника, формулирующего потребность
  - блокер: какие действия из "как в Jira" действительно нужны?
    - Почему открыт: ссылка на Jira не задает границы объема и не позволяет выделить обязательные сценарии
    - Где уточнить: у собеседника, формулирующего потребность
  - важный: по каким критериям пользователь считает процесс удобным?
    - Почему открыт: оценка "удобно" дана как абстракция без наблюдаемых критериев
    - Где уточнить: у собеседника или владельца процесса

#### Что показывает пример
- Если вход начинается с аналогии к готовому продукту, скилл не должен выдумывать требования.
- Правильный результат здесь не "догадаться", а вскрыть отсутствие фундамента.
