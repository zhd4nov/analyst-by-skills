# story-quality-reviewer

## Назначение
Проверяет и улучшает пользовательские истории.

## Вход
- список пользовательских историй

## Выход
- обновлённые истории
- статус готовности для каждой

## Контракт входа
- `Пользовательские истории`
- `Спецификация` (опционально)

## Контракт выхода
- `Пользовательские истории`
- `Статус готовности историй`
- `Открытые вопросы` (опционально)

## Сценарии сбоя
- статус готовности не объяснен или не связан с конкретной историей
- проблемы истории перечислены неявно
- проверяющий скилл исправляет историю, но не сохраняет открытые вопросы
- критичные пробелы не отражены в статусе готовности

## Правила повторного вызова
Повторный вызов нужен, если:
- хотя бы у одной истории нет статуса готовности
- статус готовности не подкреплен проверяемыми замечаниями
- критичные пробелы не вынесены в `Открытые вопросы`

При повторном вызове нужно передавать:
- текущие `Пользовательские истории`
- `Спецификация` (если есть)
- список обнаруженных проблем

## Проверки

- есть актор
- есть ценность
- есть критерии приемки
- есть `Источник`
- история независима
- нет критичных пробелов

## Статус готовности
- Высокий
- Средний
- Низкий

## Примеры

### Пример 1: История выглядит правдоподобно, но еще не готова

#### Вход
- `Пользовательские истории`:
  - ID: US-07
    - Название: Управление возвратом
    - Актор: Клиент
    - Потребность: управлять возвратом
    - Ценность: экономить время
    - Сценарий: клиент работает с возвратом
    - Критерии приемки: возвратом можно управлять

#### Ожидаемый выход
- `Пользовательские истории`:
  - ID: US-07
    - Название: Просмотр статуса возврата
    - Актор: Клиент
    - Потребность: видеть текущий статус возврата
    - Ценность: понимать состояние возврата без обращения в поддержку
    - Сценарий: клиент открывает возврат и видит текущий статус
    - Критерии приемки:
      - клиент видит текущий статус возврата
      - статус соответствует последнему зафиксированному состоянию
      - если возврат не найден, показывается понятное сообщение
- `Статус готовности историй`:
  - US-07: Средний
- `Открытые вопросы`:
  - нужно ли клиенту отменять возврат самостоятельно?

#### Что показывает пример
- Проверяющий скилл должен сужать расплывчатую историю до проверяемой ценности.
- Хорошие требования начинаются с точности формулировки, а не с широкой формулы "управлять всем".

### Пример 2: Низкий статус готовности сигнализирует о слабом фундаменте

#### Вход
- `Пользовательские истории`:
  - ID: US-11
    - Название: Экспорт заявок
    - Актор: Менеджер
    - Потребность: экспортировать заявки
    - Ценность: удобно работать
    - Сценарий: менеджер экспортирует заявки
    - Критерии приемки: файл скачивается

#### Ожидаемый выход
- `Статус готовности историй`:
  - US-11: Низкий
- `Открытые вопросы`:
  - какие данные должны попадать в экспорт?
  - для какого бизнес-решения нужен экспорт?
  - какие ограничения по формату файла обязательны?

#### Что показывает пример
- Иногда проблему нельзя "починить" только редактированием истории.
- Низкий статус готовности означает, что надо вернуться назад к требованиям, а не идти дальше в разработку.
