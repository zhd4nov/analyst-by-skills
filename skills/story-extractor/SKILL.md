# story-extractor

## Назначение
Извлекает пользовательские истории из спецификации.

## Вход
- спецификация

## Выход
- список пользовательских историй

## Контракт входа
- `Спецификация`
- `Допущения` (опционально)
- `Открытые вопросы` (опционально)

## Контракт выхода
- `Пользовательские истории`

## Сценарии сбоя
- одна история объединяет несколько разных сценариев
- история не содержит актора, потребности или ценности
- критерии приемки отсутствуют
- зависимости между историями избыточны
- история не отражает самостоятельную ценность
- у истории отсутствует явный источник в спецификации
- зависимость сформулирована абстрактно и не указывает, от чего именно зависит история

## Правила повторного вызова
Повторный вызов нужен, если:
- истории не проходят базовую структурную проверку
- одна история покрывает более одного независимого сценария
- зависимости можно уменьшить без потери смысла
- источник истории не может быть связан со спецификацией

При повторном вызове нужно передавать:
- `Спецификация`
- текущие `Пользовательские истории`
- список проблем по конкретным историям

## Правила

1. Каждая история — самостоятельная ценность
2. Не объединять разные сценарии в одну историю
3. Минимизировать зависимости
4. Для каждой истории указывать `Источник`
5. Если открытый вопрос влияет на полноту истории, сохранять его в самой истории
6. Если зависимостей или открытых вопросов нет, указывать `нет`

## Формат истории

- ID
- Название
- Источник
- Актор
- Потребность
- Ценность
- Сценарий
- Критерии приемки
- Бизнес-правила
- Зависимости
- Открытые вопросы

## Примеры

### Пример 1: Из спецификации в самостоятельные истории

#### Вход
- `Спецификация`:
  - Контекст / Проблема: клиенты не понимают, на каком этапе находится возврат
  - Бизнес-цель: снизить обращения в поддержку по статусу возврата
  - Акторы: Клиент, Оператор склада
  - Пользовательские сценарии: клиент отслеживает возврат, оператор обновляет статус возврата
  - Бизнес-правила: возврат доступен только для оплаченного заказа

#### Ожидаемый выход
- `Пользовательские истории`:
  - ID: US-01
    - Название: Просмотр статуса возврата
    - Источник:
      - Пользовательский сценарий: клиент отслеживает возврат
      - Бизнес-правило: возврат доступен только для оплаченного заказа
    - Актор: Клиент
    - Потребность: видеть текущий статус возврата
    - Ценность: понимать, что происходит с возвратом без обращения в поддержку
    - Сценарий: клиент открывает карточку возврата и видит текущий статус
    - Критерии приемки: статус отображается для существующего возврата
    - Бизнес-правила: возврат относится к оплаченному заказу
    - Зависимости: внешняя система статусов
    - Открытые вопросы: нет
  - ID: US-02
    - Название: Обновление статуса возврата
    - Источник:
      - Пользовательский сценарий: оператор обновляет статус возврата
      - Бизнес-правило: статусы допустимы только из утвержденного списка
    - Актор: Оператор склада
    - Потребность: менять статус возврата после обработки
    - Ценность: поддерживать для клиента актуальную информацию
    - Сценарий: оператор меняет статус после приемки товара
    - Критерии приемки: новый статус сохраняется и доступен клиенту
    - Бизнес-правила: статусы допустимы только из утвержденного списка
    - Зависимости: нет
    - Открытые вопросы: нет

#### Что показывает пример
- Истории должны выделяться из сценариев и ценности, а не из экранов и кнопок.
- Если начать "сначала", истории получаются независимыми и проверяемыми.

### Пример 2: Нельзя склеивать разные потребности в одну историю

#### Вход
- `Спецификация`:
  - Пользовательские сценарии: менеджер ищет заявку, назначает исполнителя, закрывает заявку, смотрит отчет

#### Неправильный выход
- Одна история: "Как менеджер, я хочу управлять заявками, чтобы работать эффективнее"

#### Правильный выход
- Несколько историй:
  - поиск заявки
  - назначение исполнителя
  - закрытие заявки
  - просмотр отчета

#### Что показывает пример
- Скилл не должен превращать несколько решений и сценариев в одну "удобную" историю.
- Если фундамент сценариев разный, истории тоже должны быть разными.
- История должна быть связана с конкретным местом в спецификации, а не появляться сама по себе.
- Зависимость должна описывать конкретную связь, а не общий риск или размытую техническую оговорку.
