# story-extractor

## Назначение
Преобразует спецификацию в минимальный набор самостоятельных пользовательских историй с проверяемой ценностью и явным источником.

## Вход
- спецификация

## Выход
- пользовательские истории
- открытые вопросы, если истории нельзя полноценно извлечь без домыслов

## Контракт входа
- `Спецификация`
- `Лог уточнений` (опционально)
- `Допущения` (опционально)
- `Открытые вопросы` (опционально)

## Контракт выхода
- `Пользовательские истории`
- `Открытые вопросы` (опционально)

## Сценарии сбоя
- одна история объединяет несколько разных пользовательских результатов
- история выделена по экрану, кнопке, полю или техническому действию, а не по пользовательской ценности
- история не содержит актора, потребности, ценности или проверяемых критериев приемки
- история не имеет явного источника в спецификации
- история опирается на факт, отсутствующий во `Входе`, но источник не позволяет связать его со `Спецификацией` или `Логом уточнений`
- история склеивает разные правила, разные развилки сценария или разные условия приемки
- зависимости описаны общими словами и не указывают конкретную блокирующую предпосылку
- открытый вопрос влияет на границы истории, но не вынесен явно

## Правила повторного вызова
Повторный вызов нужен, если:
- истории не проходят структурную проверку
- хотя бы одна история покрывает более одного результата
- история построена вокруг решения, а не вокруг потребности пользователя
- зависимости можно убрать или сузить без потери смысла
- источник истории не удается связать с конкретным местом спецификации
- для полноты истории нужны новые `Открытые вопросы`

При повторном вызове нужно передавать:
- `Спецификация`
- текущие `Пользовательские истории`
- список проблем по конкретным историям

## Правила

1. Извлекать истории только из явно зафиксированного содержания `Спецификации`.
2. Одна история = один актор + одна потребность + одна самостоятельная ценность.
3. Разделять истории, если различаются актор, результат, бизнес-правило, триггер или критерии приемки.
4. Не создавать истории по экрану, кнопке, полю, API, таблице, джобе или внутренней задаче, если за ними нет отдельной пользовательской ценности.
5. Для каждой истории указывать `Источник` с конкретной привязкой к элементам спецификации.
6. `Сценарий` описывает только основной путь получения ценности. Без дизайна, реализации и лишних деталей.
7. `Критерии приемки` должны быть наблюдаемыми, проверяемыми и независимыми от реализации.
8. `Бизнес-правила` включать только те, которые реально ограничивают историю.
9. `Зависимости` указывать только для блокирующих предпосылок. Если их нет, писать `нет`.
10. `Открытые вопросы` включать только если они влияют на полноту, границы или приемку истории. Если вопросов нет, писать `нет`.
11. Если ценность или источник нельзя сформулировать без домысла, не маскировать пробел. Вернуть `Открытые вопросы`.
12. На выходе должен быть минимальный набор историй, покрывающий спецификацию без дублирования.
13. Разрешено использовать `Альтернативные`, `Исключительные` сценарии и `Валидационные правила` как `Источник`, если они реально ограничивают границы, поведение или приемку истории.
14. Если критичная история зависит от исключительного сценария или правила валидации, их нельзя игнорировать при формулировке критериев приемки.
15. Если источник истории опирается на факт, полученный из ответа пользователя, источник должен ссылаться на соответствующее место `Спецификации` или запись в `Логе уточнений`.

## Формат истории

- ID
- Название
- Источник
- Актор
- Потребность
- Ценность
- Сценарий
- Критерии приемки
- Бизнес-правила
- Зависимости
- Открытые вопросы

## Правила разбиения

- Делить по разным акторам.
- Делить по разным пользовательским результатам.
- Делить по разным наборам бизнес-правил.
- Делить по разным условиям приемки.
- Не делить по шагам одного сценария, если ценность одна.
- Не делить по техническим слоям реализации.

## Примеры

### Пример 1: Из спецификации в самостоятельные истории

#### Вход
- `Спецификация`:
  - Контекст / Проблема: клиенты не понимают, на каком этапе находится возврат
  - Бизнес-цель: снизить обращения в поддержку по статусу возврата
  - Акторы: Клиент, Оператор склада
  - Пользовательские сценарии: клиент отслеживает возврат, оператор обновляет статус возврата
  - Бизнес-правила: возврат доступен только для оплаченного заказа

#### Ожидаемый выход
- `Пользовательские истории`:
  - ID: US-01
    - Название: Просмотр статуса возврата
    - Источник:
      - Пользовательский сценарий: клиент отслеживает возврат
      - Бизнес-правило: возврат доступен только для оплаченного заказа
    - Актор: Клиент
    - Потребность: видеть текущий статус возврата
    - Ценность: понимать, что происходит с возвратом без обращения в поддержку
    - Сценарий: клиент открывает карточку возврата и видит текущий статус
    - Критерии приемки: статус отображается для существующего возврата
    - Бизнес-правила: возврат относится к оплаченному заказу
    - Зависимости: внешняя система статусов
    - Открытые вопросы: нет
  - ID: US-02
    - Название: Обновление статуса возврата
    - Источник:
      - Пользовательский сценарий: оператор обновляет статус возврата
      - Бизнес-правило: статусы допустимы только из утвержденного списка
    - Актор: Оператор склада
    - Потребность: менять статус возврата после обработки
    - Ценность: поддерживать для клиента актуальную информацию
    - Сценарий: оператор меняет статус после приемки товара
    - Критерии приемки: новый статус сохраняется и доступен клиенту
    - Бизнес-правила: статусы допустимы только из утвержденного списка
    - Зависимости: нет
    - Открытые вопросы: нет

#### Что показывает пример
- Истории выделяются по пользовательскому результату, а не по элементам интерфейса.
- Для каждой истории есть отдельная ценность, отдельный источник и свой набор условий приемки.

### Пример 2: Нельзя склеивать разные потребности в одну историю

#### Вход
- `Спецификация`:
  - Пользовательские сценарии: менеджер ищет заявку, назначает исполнителя, закрывает заявку, смотрит отчет

#### Неправильный выход
- Одна история: "Как менеджер, я хочу управлять заявками, чтобы работать эффективнее"

#### Правильный выход
- Несколько историй:
  - поиск заявки
  - назначение исполнителя
  - закрытие заявки
  - просмотр отчета

#### Что показывает пример
- Разные результаты требуют разных историй.
- История не должна быть контейнером для целого процесса.
- Источник должен указывать, откуда именно история появилась.
- Зависимость должна быть конкретной и блокирующей.

### Пример 3: Если основание слабое, надо вернуть вопрос, а не выдумывать историю

#### Вход
- `Спецификация`:
  - Функциональное требование: пользователь может настроить уведомления
  - Контекст не уточняет, кто именно настраивает уведомления, какие каналы доступны и для какого события это нужно

#### Ожидаемый выход
- `Пользовательские истории`:
  - ID: US-01
    - Название: Настройка уведомлений
    - Источник:
      - Функциональное требование: пользователь может настроить уведомления
    - Актор: пользователь
    - Потребность: настраивать уведомления
    - Ценность: получать релевантные уведомления
    - Сценарий: пользователь меняет параметры уведомлений
    - Критерии приемки:
      - пользователь может сохранить настройки уведомлений
    - Бизнес-правила: нет
    - Зависимости: решение по доступным каналам уведомлений
    - Открытые вопросы:
      - какие каналы уведомлений входят в объем?
      - для каких событий пользователь настраивает уведомления?

#### Что показывает пример
- Скилл может извлечь черновую историю только из явно имеющегося основания.
- Недостающие границы истории должны стать `Открытыми вопросами`.
- Нельзя домысливать каналы, события или роли, если их нет в спецификации.
