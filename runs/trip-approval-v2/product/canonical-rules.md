# Канонические правила

Правило
ID: CR-01
Тип: данные
Формулировка: Заявка на командировку должна содержать даты, город, цель поездки и предварительный бюджет.
Источник: Вход
Использовать дальше:
- Спецификация -> Данные / Сущности
- Пользовательские истории -> создание заявки

Правило
ID: CR-02
Тип: переход состояния
Формулировка: После отправки заявка должна попасть руководителю на согласование.
Источник: Вход
Использовать дальше:
- Спецификация -> Модель состояний / Переходы
- Пользовательские истории -> отправка заявки на согласование

Правило
ID: CR-03
Тип: бизнес-правило
Формулировка: Руководитель может согласовать или отклонить заявку.
Источник: Вход
Использовать дальше:
- Спецификация -> Бизнес-правила
- Пользовательские истории -> рассмотрение заявки руководителем

Правило
ID: CR-04
Тип: валидация
Формулировка: При отклонении заявки руководитель должен указать причину.
Источник: Вход
Использовать дальше:
- Спецификация -> Валидационные правила
- Пользовательские истории -> рассмотрение заявки руководителем

Правило
ID: CR-05
Тип: видимость
Формулировка: Сотрудник должен видеть статус заявки.
Источник: Вход
Использовать дальше:
- Спецификация -> Функциональные требования
- Пользовательские истории -> просмотр статуса заявки

Правило
ID: CR-06
Тип: видимость
Формулировка: После согласования руководителем заявка должна быть доступна бухгалтерии.
Источник: Вход
Использовать дальше:
- Спецификация -> Бизнес-правила
- Пользовательские истории -> подготовка документов бухгалтерией

Правило
ID: CR-07
Тип: бизнес-правило
Формулировка: Бухгалтерия не согласует поездку.
Источник: Вход
Использовать дальше:
- Спецификация -> Бизнес-правила
- Пользовательские истории -> подготовка документов бухгалтерией

Правило
ID: CR-08
Тип: переход состояния
Формулировка: Бухгалтерия должна отмечать, что подготовка аванса и документов завершена.
Источник: Вход
Использовать дальше:
- Спецификация -> Модель состояний / Переходы
- Пользовательские истории -> подготовка документов бухгалтерией

Правило
ID: CR-09
Тип: ограничение
Формулировка: После отправки сотрудник не должен незаметно менять заявку.
Источник: Вход
Использовать дальше:
- Спецификация -> Ограничения
- Пользовательские истории -> отмена заявки для изменения

Правило
ID: CR-10
Тип: ограничение
Формулировка: В первой версии не входят бронирование билетов, интеграция с календарем, расчет суточных и хранение чеков.
Источник: Вход
Использовать дальше:
- Спецификация -> Границы объема

Правило
ID: CR-11
Тип: бизнес-правило
Формулировка: После отправки заявку нельзя редактировать; для изменения дат, города, цели поездки или бюджета сотрудник отменяет текущую заявку и создает новую.
Источник: Лог уточнений: CL-01
Использовать дальше:
- Спецификация -> Бизнес-правила
- Пользовательские истории -> отмена заявки для изменения

Правило
ID: CR-12
Тип: переход состояния
Формулировка: Отмененная заявка больше не участвует в согласовании и подготовке документов.
Источник: Лог уточнений: CL-01
Использовать дальше:
- Спецификация -> Модель состояний / Переходы
- Пользовательские истории -> отмена заявки для изменения

Правило
ID: CR-13
Тип: переход состояния
Формулировка: Новая заявка, созданная вместо отмененной, проходит процесс согласования заново.
Источник: Лог уточнений: CL-01
Использовать дальше:
- Спецификация -> Модель состояний / Переходы
- Пользовательские истории -> создание заявки
