# Пользовательские истории

История
ID: US-01
Название: Создание заявки на командировку
Источник
- Спецификация -> Функциональные требования: создание заявки с датами, городом, целью поездки и предварительным бюджетом.
- Спецификация -> Данные / Сущности: заявка на командировку.
- Канонические правила: CR-01.
Актор: Сотрудник.
Потребность: создать заявку на командировку с ключевыми параметрами поездки.
Ценность: зафиксировать данные поездки в едином процессе вместо переписки.
Сценарий: сотрудник создает заявку и указывает даты, город, цель поездки и предварительный бюджет.
Критерии приемки
- Сотрудник может создать заявку с датами, городом, целью поездки и предварительным бюджетом.
- Без дат, города, цели поездки или предварительного бюджета заявка не считается готовой к отправке.
- Созданная заявка имеет статус Черновик.
Бизнес-правила
- Заявка на командировку содержит даты, город, цель поездки и предварительный бюджет.
Зависимости
- Нет.
Открытые вопросы
- Нет.

История
ID: US-02
Название: Отправка заявки на согласование руководителю
Источник
- Спецификация -> Пользовательские сценарии: сотрудник отправляет заявку руководителю на согласование.
- Спецификация -> Модель состояний / Переходы: Черновик -> На согласовании.
- Канонические правила: CR-02, CR-09.
Актор: Сотрудник.
Потребность: отправить заполненную заявку руководителю.
Ценность: запустить формальный процесс согласования без переписки.
Сценарий: сотрудник отправляет заполненную заявку, после чего она становится доступна руководителю для решения.
Критерии приемки
- Заполненная заявка может быть отправлена руководителю.
- После отправки заявка получает статус На согласовании.
- После отправки заявка недоступна сотруднику для редактирования.
Бизнес-правила
- После отправки заявка переходит руководителю на согласование.
- В первой версии после отправки заявку нельзя редактировать.
Зависимости
- Заявка создана и содержит обязательные данные.
Открытые вопросы
- Нет.

История
ID: US-03
Название: Согласование заявки руководителем
Источник
- Спецификация -> Функциональные требования: руководитель согласует заявку.
- Спецификация -> Модель состояний / Переходы: На согласовании -> Согласована.
- Канонические правила: CR-03, CR-06.
Актор: Руководитель.
Потребность: согласовать заявку на командировку.
Ценность: подтвердить поездку и передать ее в следующий операционный этап.
Сценарий: руководитель рассматривает заявку в статусе На согласовании и согласует ее.
Критерии приемки
- Руководитель может согласовать заявку в статусе На согласовании.
- После согласования заявка получает статус Согласована.
- Согласованная заявка становится доступной бухгалтерии.
Бизнес-правила
- Руководитель может согласовать или отклонить заявку.
- После согласования руководителем заявка должна быть доступна бухгалтерии.
Зависимости
- Заявка находится в статусе На согласовании.
Открытые вопросы
- Нет.

История
ID: US-04
Название: Отклонение заявки руководителем
Источник
- Спецификация -> Альтернативные сценарии: руководитель отклоняет заявку и указывает причину отклонения.
- Спецификация -> Валидационные правила: отклонение без причины недопустимо.
- Канонические правила: CR-03, CR-04.
Актор: Руководитель.
Потребность: отклонить заявку с объяснением причины.
Ценность: зафиксировать понятное решение по заявке для сотрудника.
Сценарий: руководитель отклоняет заявку в статусе На согласовании и указывает причину.
Критерии приемки
- Руководитель может отклонить заявку в статусе На согласовании.
- Система не принимает отклонение без причины.
- После отклонения заявка получает статус Отклонена.
- Причина отклонения сохраняется и доступна сотруднику вместе со статусом.
Бизнес-правила
- Руководитель может согласовать или отклонить заявку.
- При отклонении заявки причина обязательна.
Зависимости
- Заявка находится в статусе На согласовании.
Открытые вопросы
- Нет.

История
ID: US-05
Название: Просмотр статуса заявки сотрудником
Источник
- Спецификация -> Функциональные требования: система показывает сотруднику статус заявки.
- Канонические правила: CR-05.
Актор: Сотрудник.
Потребность: видеть текущий статус своей заявки.
Ценность: понимать, на каком этапе находится командировка без уточнений в почте или мессенджере.
Сценарий: сотрудник открывает свою заявку и видит ее текущий статус.
Критерии приемки
- Сотрудник видит статус созданной им заявки.
- Статус отражает текущее состояние заявки: Черновик, На согласовании, Согласована, Отклонена, Отменена или Документы подготовлены.
- Для отклоненной заявки сотрудник видит причину отклонения.
Бизнес-правила
- Сотрудник должен видеть статус своей заявки.
Зависимости
- Заявка существует.
Открытые вопросы
- Нет.

История
ID: US-06
Название: Отмена отправленной заявки вместо редактирования
Источник
- Спецификация -> Альтернативные сценарии: сотрудник отменяет отправленную заявку и создает новую.
- Спецификация -> Ограничения: после отправки заявку нельзя редактировать.
- Лог уточнений: CL-01.
- Канонические правила: CR-09, CR-10, CR-11.
Актор: Сотрудник.
Потребность: отменить отправленную заявку, если нужно изменить даты, город, цель или бюджет.
Ценность: изменить намерение по поездке без незаметного изменения уже отправленной заявки.
Сценарий: сотрудник отменяет текущую отправленную заявку и при необходимости создает новую заявку с корректными данными.
Критерии приемки
- Сотрудник может отменить отправленную заявку.
- После отмены заявка получает статус Отменена.
- Отмененная заявка больше не участвует в согласовании.
- Отмененная заявка больше не участвует в подготовке документов.
- Система не позволяет редактировать даты, город, цель или бюджет после отправки заявки.
Бизнес-правила
- В первой версии после отправки заявку нельзя редактировать.
- Если сотруднику нужно изменить даты, город, цель или бюджет после отправки, он отменяет текущую заявку и создает новую.
- Отмененная заявка больше не участвует в согласовании и подготовке документов.
Зависимости
- Заявка уже отправлена.
Открытые вопросы
- Нет.

История
ID: US-07
Название: Просмотр согласованных заявок бухгалтерией
Источник
- Спецификация -> Функциональные требования: согласованная заявка доступна бухгалтерии.
- Спецификация -> Бизнес-правила: бухгалтерия не согласует поездку.
- Канонические правила: CR-06, CR-07.
Актор: Бухгалтерия.
Потребность: видеть согласованные заявки на командировку.
Ценность: подготовить аванс и документы по подтвержденным поездкам без дополнительных уточнений в переписке.
Сценарий: бухгалтерия просматривает заявки, которые руководитель согласовал.
Критерии приемки
- Бухгалтерия видит заявки в статусе Согласована.
- Бухгалтерия не согласует и не отклоняет поездку.
- Отмененные и отклоненные заявки не участвуют в подготовке документов.
Бизнес-правила
- После согласования руководителем заявка должна быть доступна бухгалтерии.
- Бухгалтерия не согласует поездку.
- Отмененная заявка больше не участвует в подготовке документов.
Зависимости
- Руководитель согласовал заявку.
Открытые вопросы
- Нет.

История
ID: US-08
Название: Отметка завершения подготовки документов
Источник
- Спецификация -> Функциональные требования: бухгалтерия отмечает завершение подготовки документов.
- Спецификация -> Модель состояний / Переходы: Согласована -> Документы подготовлены.
- Канонические правила: CR-08.
Актор: Бухгалтерия.
Потребность: отметить, что аванс и документы по согласованной заявке подготовлены.
Ценность: зафиксировать завершение бухгалтерского этапа процесса.
Сценарий: бухгалтерия отмечает завершение подготовки документов по согласованной заявке.
Критерии приемки
- Бухгалтерия может отметить завершение подготовки документов только по согласованной заявке.
- После отметки заявка получает статус Документы подготовлены.
- Сотрудник видит обновленный статус заявки.
Бизнес-правила
- Бухгалтерия может отметить завершение подготовки документов по согласованной заявке.
Зависимости
- Заявка находится в статусе Согласована.
Открытые вопросы
- Нет.
